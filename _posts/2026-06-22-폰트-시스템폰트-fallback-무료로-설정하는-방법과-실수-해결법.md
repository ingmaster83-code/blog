# 폰트 시스템폰트 fallback 무료 설정, 이 실수만 피워도 사이트 속도가 달라집니다

"웹폰트를 적용했는데 왜 특정 기기에서 글자가 네모로 나오지?" "구글 폰트 로딩이 느려서 사이트 이탈률이 높아졌어요." 이런 고민, 한 번쯤 해보셨죠?

저도 처음 웹사이트를 운영할 때 폰트 시스템폰트 fallback을 제대로 설정하지 않아 방문자에게 깨진 화면을 보여준 경험이 있습니다. 특히 한국어 웹사이트는 한글 폰트 용량이 커서 로딩 지연이 발생하기 쉬운데, 적절한 fallback 폰트를 지정하지 않으면 사용자 경험이 크게 떨어집니다.

오늘은 제가 직접 겪은 실수와 해결법을 바탕으로, **폰트 시스템폰트 fallback 무료** 도구인 FontKit을 소개해드리려고 합니다.

---

## FontKit: fallback 폰트 조합을 무료로 자동 생성하는 도구

제가 찾은 해결책은 바로 [FontKit 무료로 사용하기](https://fontkit.wooahouse.com)입니다. 이 도구는 웹폰트와 시스템폰트를 조합해 최적의 CSS fallback 스택을 자동으로 추천해줍니다.

FontKit의 가장 큰 장점은 복잡한 코드 없이도 윈도우, 맥OS, iOS, 안드로이드 등 주요 운영체제에 맞는 fallback 폰트 순서를 한 번에 생성한다는 점입니다. 직접 OS별 시스템폰트를 조사하고 테스트할 필요가 없어 시간을 크게 절약할 수 있습니다.

---

## FontKit 주요 기능 상세 분석

### 1. 웹폰트와 시스템폰트 자동 매칭

FontKit에 접속하면 가장 먼저 사용하려는 웹폰트를 선택합니다. 그러면 도구가 자동으로 해당 폰트와 유사한 시스템폰트를 찾아 fallback 리스트를 구성해줍니다. 예를 들어 'Noto Sans KR'을 선택하면 Windows용 '맑은 고딕', Mac용 '애플 산돌고딕 네오', iOS용 'SF Pro' 등을 순서대로 배치합니다.

### 2. OS별 폰트 fallback 테스트

직접 여러 기기에서 폰트가 어떻게 보일지 미리 확인할 수 있는 기능이 있습니다. 폰트 시스템폰트 fallback을 설정할 때 가장 어려운 점이 OS별 렌더링 차이인데, 이 기능덕분에 실제 배포 전에 문제를 발견하고 수정할 수 있습니다.

### 3. CSS 코드 자동 생성

설정이 끝나면 바로 복사해서 사용할 수 있는 CSS 코드를 제공합니다. `font-family: 'Noto Sans KR', 'Malgun Gothic', 'Apple SD Gothic Neo', sans-serif;` 이런 형태로 완성된 코드를 그대로 프로젝트에 붙여넣기만 하면 됩니다.

---

## 실제 사용 후기: 3가지 실수와 해결법

### 실수 1: 폰트 시스템폰트 fallback 순서를 무시했다

처음에는 그냥 'sans-serif' 하나만 넣었습니다. 결과는 처참했습니다. 맥에서 웹폰트가 로딩되기 전까지 기본 시스템폰트가 표시되는데, 그 모양이 너무 달라서 레이아웃이 흔들렸습니다.

**해결법:** FontKit으로 생성한 **시스템폰트 fallback 리스트**를 적용한 후, 웹폰트 로딩 전후의 폰트 높이 차이가 거의 사라졌습니다. 특히 x-height(소문자 높이)가 비슷한 폰트끼리 매칭해주는 기능이 효과적이었습니다.

### 실수 2: 한글 폰트만 신경 쓰고 영문 폰트 fallback을 놓쳤다

한글 웹폰트는 잘 설정했지만, 영문과 숫자에 대한 폰트 시스템폰트 fallback을 따로 지정하지 않았습니다. 그 결과 영문이 예상과 다른 글꼴로 표시되어 디자인 통일성이 깨졌습니다.

**해결법:** FontKit은 한글과 영문 폰트를 분리해서 fallback 설정을 할 수 있습니다. `font-family: 'Noto Sans KR', 'Malgun Gothic', -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif;` 이렇게 영문 시스템폰트를 앞에 배치하는 방식을 추천해주어 문제를 해결했습니다.

### 실수 3: 구글 폰트 로딩 속도만 고려하고 fallback을 생략했다

웹폰트 로딩 속도를 높이기 위해 `font-display: swap`만 사용하고 fallback을 신경 쓰지 않았습니다. 그 결과 폰트가 로딩되는 순간 레이아웃 시프트(CLS)가 발생해 구글 코어 웹 바이탈 점수가 나빠졌습니다.

**해결법:** FontKit이 생성한 **웹폰트 최적화 코드**를 적용하면서 `font-display: optional`과 함께 정확한 fallback을 사용했습니다. 로딩 속도와 레이아웃 안정성 모두 개선되어 Lighthouse 성능 점수가 15점 이상 올랐습니다.
