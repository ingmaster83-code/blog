## nginx 설정 파일, 이 실수로 서버가 죽었습니다

몇 주 전, 운영 중인 웹 서비스에 리버스 프록시 설정을 추가하려고 `/etc/nginx/conf.d/default.conf`를 직접 수정했습니다. 세미콜론 하나를 빼먹고 `nginx -s reload`를 실행했는데, 서버가 아예 응답하지 않았습니다. 결국 30분 동안 원복 작업을 하며 식은땀을 흘렸죠. 이런 실수는 nginx를 다루는 개발자라면 누구나 한 번쯤 겪습니다. 특히 location 블록의 중첩 규칙이나 allow/deny 지시어 순서를 잘못 쓰면 예상치 못한 403 에러가 발생하기 쉽습니다.

이 글에서는 nginx 설정 파일 무료 방법을 찾는 분들을 위해, 설치 없이 브라우저만으로 실수를 방지하고 설정을 완성하는 도구를 소개합니다.

## 단계별 사용법: 30초 만에 설정 완료하기

### ### 1단계: 설정 파일 준비
먼저 서버의 nginx 설정 파일을 복사합니다. 보통 `/etc/nginx/nginx.conf` 또는 `/etc/nginx/conf.d/` 디렉토리 아래에 있습니다. `cat /etc/nginx/nginx.conf` 명령어로 내용을 복사하거나, SCP로 로컬에 다운로드하세요.

### ### 2단계: WooaDev에 붙여넣기
[WooaDev 페이지](https://wooadev.wooahouse.com)에 접속하여 좌측 편집기에 복사한 설정 내용을 붙여넣습니다. 이때 주석도 함께 포함해도 됩니다.

### ### 3단계: 문법 검증 실행
"설정 검증" 버튼을 클릭하면 즉시 구문 분석이 시작됩니다. 일반적인 실수인 **세미콜론 누락, 중괄호 불일치, 잘못된 지시어 사용** 등을 빨간색으로 표시해줍니다. 예를 들어 `proxy_pass` 뒤에 `[http://`를](http://`를) 빼먹는 실수도 잡아냅니다.

### ### 4단계: 수정 및 재검증
오류가 표시된 부분을 클릭하면 해당 라인으로 자동 포커스됩니다. 수정 후 다시 "검증" 버튼을 눌러 초록색 성공 메시지를 확인하세요. 서버 설정의 민감성을 고려해 **반드시 2회 이상 검증**하는 것을 추천합니다.

### ### 5단계: 적용 (복사 후 서버에 반영)
검증이 완료된 설정을 복사하여 서버의 원본 파일을 덮어씁니다. 그 후 `sudo nginx -s reload`로 적용하면 됩니다. 이 도구는 클립보드 자동 복사 기능도 지원하므로 편리합니다.

## 주의사항: 설정 파일 수정 시 반드시 체크할 점

nginx 설정 파일을 다룰 때 가장 흔한 실수는 **순서 의존성**을 무시하는 것입니다. 예를 들어 location 블록 내에서 `try_files` 지시어는 반드시 `proxy_pass`보다 먼저 와야 합니다. 또한 `server_name`에 와일드카드(`*.example.com`)를 사용할 때는 반드시 정확한 도메인이 먼저 매칭되도록 순서를 조정해야 합니다.

다른 주의점은 **include 구문**입니다. `include /etc/nginx/conf.d/*.conf;` 같은 구문이 있는 경우, 다른 파일과 충돌하지 않는지 확인해야 합니다. WooaDev는 include된 파일까지 검증하지는 못하므로, 개별 파일을 따로 검증하거나 전체 설정을 하나의 파일로 합쳐서 검증하는 방법을 권장합니다.

마지막으로, 적용 전 반드시 백업을 하세요. `cp /etc/nginx/nginx.conf /etc/nginx/nginx.conf.backup` 명령어로 원본을 저장해두면 혹시 모를 상황에 대비할 수 있습니다.
