# git 명령어 치트 시트 온라인, 실수 없이 바로 찾는 법

어제도 나는 `git reset --hard`를 잘못 쳐서 로컬 작업 내역을 통째로 날렸다. 아직 커밋도 안 한 상태였기에 되돌릴 방법도 없었다. 터미널 앞에서 멍하니 "아, 또?"라는 말만 반복했다. git을 쓰는 개발자라면 누구나 이런 순간을 한 번쯤 겪었을 것이다. 명령어가 많고 비슷비슷해서 자주 헷갈리기 때문이다. 그래서 준비했다. **git 명령어 치트 시트 온라인** 하나면 이런 실수를 반복하지 않는다.

## WooaDev가 제공하는 git 명령어 치트 시트의 핵심 기능

### 1️⃣ 실수 유발 명령어를 한눈에 비교

`git revert`와 `git reset`의 차이를 현장에서 바로 확인할 수 있다. 대부분의 실수는 `--soft`, `--mixed`, `--hard` 옵션을 잘못 선택할 때 발생하는데, 이 도구는 각 커맨드의 실행 결과를 예시와 함께 보여준다. 예를 들어 "되돌리기" 섹션에서는 커밋을 취소하는 4가지 방법과 각각의 위험도를 색상으로 표시해준다. 더 이상 구글에서 "git undo commit"을 검색할 필요가 없다.

### 2️⃣ 브랜치 작업에서 자주 틀리는 패턴

협업할 때 가장 많이 헷갈리는 것이 브랜치 명령어다. `git checkout -b`와 `git switch -c`의 차이, 그리고 `git merge`와 `git rebase`의 사용 타이밍을 명확하게 구분해준다. 특히 "충돌 해결 시나리오" 섹션은 실제 프로젝트에서 브랜치 병합 중 충돌이 났을 때 어떤 명령어를 순서대로 입력해야 하는지 단계별로 정리되어 있다. 초보자도 이 치트 시트만 보면 깃허브 명령어 정리를 따로 외울 필요가 없다.

### 3️⃣ 원격 저장소 작업 최적화

`git push`와 `git pull`의 기본 사용법은 알지만, `--force` 옵션을 남발하다가 팀원의 코드를 덮어쓰는 참사를 막아준다. 이 도구는 원격 저장소와 싱크를 맞출 때 반드시 확인해야 할 명령어 순서를 체크리스트 형태로 제공한다. 또한 `git fetch`와 `git pull`의 차이를 실제 로그 예시와 함께 설명해주어, "최신 코드를 가져왔는데 왜 내 로컬에 반영이 안 되지?"라는 의문을 완전히 해소해준다.

### 4️⃣ 파일 상태 추적과 취소 명령어

실제로 개발자들이 가장 많이 하는 실수는 `git add` 후에 잘못된 파일을 스테이징하는 경우다. "스테이징 해제" 섹션에서는 `git restore --staged`와 `git rm --cached`의 차이를 실전 예시로 비교해준다. 또한 `git status`로 확인한 파일 상태에 따라 어떤 명령어를 써야 하는지 if-else 형식으로 정리되어 있어, 초보자도 무리 없이 따라 할 수 있다.

## 실제 사용 후기: 2주간 팀 프로젝트에서 써본 결과

나는 지난 2주 동안 4인 팀 프로젝트를 진행하면서 WooaDev를 상시 열어두고 작업했다. 효과는 극적이었다. 평소에 하루 5~6번은 `git status`를 잘못 읽거나 `git log` 옵션을 헷갈려서 검색했는데, 이제는 치트 시트에서 바로 복사해서 붙여 넣는다. 특히 팀원들이 `git rebase`를 잘못 사용해서 히스토리가 꼬이는 일이 잦았는데, 이 도구의 "브랜치 재배열" 섹션을 따라 하면서부터는 그런 문제가 완전히 사라졌다.

가장 만족스러운 점은 "당황하지 않게 해준다"는 것이다. 급하게 커밋을 잘못했을 때, `git commit --amend`나 `git reset`을 어떻게 써야 할지 패닉에 빠지곤 했는데, 이제는 차분하게 도구를 열고 해당 명령어를 찾아보면 된다. 개발 생산성이 눈에 띄게 올랐고, 팀 내 코드 리뷰에서도 "이 부분은 이렇게 고치는 게 낫다"는 식의 git 관련 피드백을 더 이상 받지 않게 되었다.

한 가지 아쉬운 점은 모바일 최적화가 덜 되어 있다는 것이다. 맥북이나 데스크톱에서는 화면 분할해서 보기 좋지만, 아이패드로 볼 때는 글자 크기가 조금 작게 느껴진다. 그래도 핵심 기능은 완벽하게 작동하므로 작업 환경이 데스크톱 위주라면 전혀 문제되지 않는다.
